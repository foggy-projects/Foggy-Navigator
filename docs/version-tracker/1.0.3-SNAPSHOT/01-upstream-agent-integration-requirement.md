# 上游 Agent 接入首版需求

## 文档作用

- doc_type: requirement
- intended_for: execution-agent | reviewer | product-owner
- purpose: 将 1.0.3-SNAPSHOT 版本“让上游真实接入并业务可用”的目标正式落盘，作为后续实现规划、接口设计和验收的上游依据

## Version

- `1.0.3-SNAPSHOT`

## Priority

- P0

## Status

- Draft
- 2026-04-15 初版记录

## Background

当前平台已经具备一定的第三方接入基础：

- 有面向第三方系统的 Open API，可发起 Agent 任务、查询任务、取消任务
- 平台内部已有统一会话管理、消息落库和 SSE 推送能力
- 已有 Java Open SDK，前端侧也有 SSE / Chat 相关封装

但这些能力仍然偏“平台内部可用”或“单点调用可用”，尚未形成一套完整、稳定、适合上游系统正式接入的业务合同。当前主要问题包括：

- 上游缺少正式可用的会话列表和会话消息读取合同
- Open API 任务轮询只能稳定获取任务状态和终态结果，不能表达“任务进行中新增了哪些消息”
- 同一个任务内 Agent 可能连续产生多条消息，但上游缺少标准方式按任务拉取这些增量消息
- Open API、内部 Session API、SSE、Worker 内部接口之间的边界尚未在文档和 SDK 层收口
- 还没有一个真实可运行的上游接入示例 Demo 来证明业务链路可用

## Problem Statement

如果继续沿用当前“发任务 + 查最终结果”的接入方式，上游只能把平台当成一个异步黑盒调用点，无法支撑更真实的业务场景，例如：

- 展示用户会话列表
- 读取某个会话中的完整消息
- 在任务执行过程中持续获取 Agent 新产生的多条消息
- 对同一任务内的中间消息、工具消息、状态消息进行跟踪与展示
- 用标准示例验证上游如何接入、如何轮询、如何回放会话

因此，1.0.3-SNAPSHOT 需要将“可接入”升级为“可用于真实业务”。

## Target Outcome

本版本目标是让上游系统能够以正式、稳定、可文档化的方式接入平台 Agent，并完成一个最小可运行的示例 Demo。

首版完成后，上游至少应能：

1. 获取可访问的会话上下文列表
2. 获取指定会话上下文中的消息列表
3. 发起新的 Agent 任务
4. 按任务轮询任务状态
5. 在任务执行过程中轮询新增消息，而不是只能等待最终结果
6. 识别同一个任务内产生的多条消息与其 `taskId` 的归属关系
7. 参考官方示例 Demo 完成基本接入

## Scope

### In Scope

- 上游会话列表能力
- 上游会话消息查询能力
- 上游任务发起、状态查询、取消能力
- 上游任务执行中的增量消息轮询能力
- `taskId` 与消息、会话、Worker 任务映射关系的正式化
- 对接文档整理
- Java SDK / 其他 SDK 的补齐评估
- 最小接入 Demo

### Out of Scope

- 大范围重构内部消息存储模型
- 首版强制提供 WebSocket
- 一次性设计所有高级产物下载、文件浏览、目录执行类外部能力
- 首版就支持所有内部事件类型完整开放

## Functional Requirements

### 1. 会话上下文列表

平台需要提供面向上游的正式会话上下文列表能力，允许上游系统加载当前用户可访问的会话列表。

首版应至少支持：

- 按用户上下文返回会话上下文列表
- 返回会话上下文基础信息
- 支持后续继续扩展筛选和分页

建议返回字段至少包括：

- `contextId`
- `agentId`
- `title`
- `status`
- `createdAt`
- `updatedAt`
- `latestTaskId`

补充约束：

- 对外只暴露 `contextId` 作为会话主键
- 内部 `sessionId` 不作为上游正式合同字段
- 一个 `contextId` 在平台内部只绑定一个 `sessionId`
- 绑定建立后不允许变更

### 2. 会话消息列表

平台需要提供面向上游的正式会话消息读取能力，允许上游系统加载某个会话上下文中的消息。

首版应至少支持：

- 按 `contextId` 获取消息列表
- 支持分页或增量加载
- 消息中显式返回 `taskId`

建议返回字段至少包括：

- `messageId`
- `contextId`
- `taskId`
- `role`
- `type`
- `content`
- `metadata`
- `createdAt`

### 3. 任务发起与状态轮询

平台继续保留当前异步任务模型，上游可发起任务并获取 `taskId`。

首版应至少支持：

- 发起任务
- 查询任务状态
- 取消任务
- 返回任务与会话的关联关系

建议任务返回至少包括：

- `taskId`
- `contextId`
- `status`
- `result`
- `errorMessage`
- `createdAt`

### 4. 任务进行中的消息轮询

平台需要补齐“按任务获取新增消息”的正式能力。

该能力是本版本核心新增目标之一。

首版应支持：

- 上游通过轮询持续获取某个任务在执行过程中新增的消息
- 同一任务中可能有多条消息，必须完整表达，而不是只返回最后一条
- 能区分是否还有更多消息
- 能避免重复返回同一批消息

建议首版采用以下任一正式模式：

1. `taskId + cursor` 轮询新增消息
2. `taskId + sinceMessageId` 轮询新增消息
3. `taskId + sinceSeq` 轮询任务事件投影

本项后续实现规划中需要在兼容性、存储成本、协议稳定性之间做取舍。

### 5. taskId 串联规则

平台需要将以下关系整理为正式契约，并在必要时对外返回：

- 平台 `taskId`
- 所属 `contextId`
- 内部 `sessionId`
- Worker 内部任务 ID
- 任务过程中产生的多条消息

首版至少要保证：

- 上游可基于平台 `taskId` 识别消息归属
- 同一任务内多条消息具备稳定归并方式
- 上游只需要理解 `contextId + taskId` 两个主键
- `contextId -> sessionId` 在内部是一对一稳定映射
- 调试层面可追溯平台 `taskId` 与 Worker 内部任务 ID 的关系

### 6. 对接文档与 SDK

本版本需要补齐正式对接材料，至少包括：

- 接口说明
- 典型接入流程
- 轮询时序说明
- 会话 / 消息 / 任务对象结构说明
- `taskId` 串联规则说明
- Demo 使用说明

同时需要评估：

- Java Open SDK 是否补齐会话与消息相关 API
- 是否需要补一个最小前端或通用 HTTP SDK 示例

### 7. 上游接入 Demo

本版本需要交付一个可运行的接入示例 Demo，用于证明：

- 上游如何创建任务
- 如何轮询状态
- 如何轮询新增消息
- 如何回放会话列表和消息列表

Demo 可以是：

- 一个最小后端示例
- 一个最小前端页面
- 或两者结合

但必须能清楚展示正式接入方式，而不是内部调试脚本拼装。

## Non-Functional Requirements

- 接口语义稳定，避免对上游暴露内部临时字段语义
- 分页或增量协议应避免重复消费问题
- 任务与消息关联规则要清晰、可追踪、可调试
- 文档必须能独立支撑第三方对接，不依赖口头解释

## Ownership

- `session-module`: 会话列表、会话消息、统一任务视图、SSE / 轮询边界
- `addons/claude-worker-agent`: Open API、Claude Worker 任务投影、taskId 与 workerTaskId 串联
- `navigator-open-sdk`: 对外 SDK 补齐
- `packages/foggy-chat` 或示例工程: Demo / 示例接入层
- `docs/version-tracker/1.0.3-SNAPSHOT/`: 版本需求与规划基线

## Acceptance Criteria

### API Contract

- 上游可通过正式接口获取会话列表
- 上游可通过正式接口获取会话消息列表
- 上游可通过正式接口发起任务、查询任务、取消任务
- 上游可通过正式接口轮询某个任务在执行过程中新增的消息
- 同一任务中多条消息的 `taskId` 归属规则清晰且可验证

### Documentation

- 有一份面向上游的统一对接文档
- 文档明确区分 Open API、内部 Session API、SSE 和 Worker 内部接口边界
- 文档包含至少一个完整接入时序示例

### Demo

- 有一个最小可运行的上游接入 Demo
- Demo 能真实跑通“创建任务 -> 获取会话/消息 -> 轮询进行中消息 -> 获取结果”

## Constraints

- 首版优先保证业务可用，不追求一次性开放所有内部能力
- 不允许仅以内部 UI 接口替代正式上游接口而不收口契约
- 不允许让 `taskId` 与消息关联规则只停留在实现细节中而没有文档化

## Open Questions

- 首版增量消息轮询是否直接对外暴露消息对象，还是先暴露任务事件投影
- 是否要在消息表层面新增显式 `taskId` 列，还是首版继续基于 metadata 承载
- Demo 的交付形态以 Java 示例、前端页面还是独立 sample 工程为主
- Java SDK 之外，是否需要同步补充脚本型示例 SDK
