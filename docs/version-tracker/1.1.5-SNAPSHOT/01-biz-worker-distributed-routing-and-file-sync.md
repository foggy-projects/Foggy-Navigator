# Business Agent Worker Pool 分布式路由与上下文同步演进规划

## 1. 背景与现状 (MVP 阶段)
在当前的 Navigator MVP 架构中，引入了 `BizWorkerPool`（业务工作池）作为承接第三方应用（如 TMS）任务的逻辑执行层。但在实际的物理执行逻辑中，目前的策略被极度简化：
- **永远选第一个节点**：由于 `LanggraphBusinessAgentWorkerTaskLauncher` 中的查询逻辑强制只拉取 `ENABLED` 状态的 Worker 并在有序集合中返回 `.findFirst()`，实际的流量分发演变成了 100% 打向该池内最早且存活的单个 Worker。
- **配置即占位**：目前控制面页面上填写的 `ROUND_ROBIN` 或 `DIRECT` 路由策略仅作为元数据占位，暂未在下层流量派发中生效。

这个极简架构（相当于 1 个 APP 默认强绑定了 1 个 Worker 节点）非常契合现阶段的诉求，它**从物理层面完全规避了分布式状态同步的难题**。用户的个人目录与上游 APP 的公共技能都在唯一的节点磁盘上，不涉及多节点文件一致性问题。

## 2. 后续迭代面临的架构挑战
当未来的并发任务量达到单节点瓶颈，或者为了高可用我们必须启动多个 `LANGGRAPH_BIZ` Worker 并真正开启负载均衡（如 ROUND_ROBIN）时，系统将面临复杂的**分布式文件共享与状态同步**问题：
1. **个人目录（状态上下文）**：用户不同轮次的请求可能落在不同的物理节点，导致由于工作区跳变引发的上下文/临时文件丢失。
2. **上游公共技能分发**：不同上游应用要求挂载不同的公共技能，集群内的 Worker 必须确保拥有（并隔离）这些最新的只读脚本。

## 3. 未来的演进方向评估

若业务增长需要真正的分布式集群化运行，建议评估以下演进路径：

### 方案一：会话黏滞路由 (Workspace Affinity) + 延迟同步（推荐演进路径）
- **核心思想**：放弃纯粹的 Round Robin，在路由层（通过一致性哈希或 Session Hash）引入黏滞机制。保证同一个用户 Session 的生命周期内流量始终打向同一台 Worker，实现对原生磁盘读写性能的 100% 利用。
- **公共技能分发**：当 Worker 节点启动或首次被分发某应用的流量时，触发 Init 流程，从服务端 OSS 动态同步最新的技能包到只读缓存区。
- **容灾处理**：依托框架底层的 FileCheckpointing 能力，任务空闲时备份关键目录状态，当节点宕机重启时通过拉取快照恢复。

### 方案二：基于对象存储的按需挂载同步 (OSS On-Demand Sync)
- **核心思想**：强制 Worker 为纯无状态架构（Stateless）。任务开始前，拉取用户的上下文快照到临时目录；执行结束后，将变更生成的 Diff 打包回传至 OSS。
- **评估**：架构干净利落，但任务启动的冷启动时间（网络 I/O 延迟）较高。

### 方案三：中心化虚拟文件网关 (Virtual FS)
- **核心思想**：Worker 本地不落地任何业务文件，通过底层劫持或 MCP 协议，将 I/O 读写透明重定向到中心的分布式存储服务。
- **评估**：对第三方原生工具（依赖真实本地 fs 如 `npm`, `git`）的兼容性差，开发成本和系统侵入性极高，暂不推荐。

## 4. 结论与下一步动作
现阶段维持当前的单节点独占派发（`.findFirst()`）逻辑。
待业务负载确实达到多节点瓶颈时，优先在 `LanggraphBusinessAgentWorkerTaskLauncher` 中重构 Worker 分发算法，将 `BizWorkerPoolEntity` 中定义的 `routingPolicy` 与黏滞路由结合起来落地。
