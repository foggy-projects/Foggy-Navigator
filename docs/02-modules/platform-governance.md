# 平台设置与资源治理

## 1. 功能定位

平台设置负责管理系统运行所依赖的外部资源与内部策略，是当前平台治理面的核心入口。

对应前端入口：

- `设置`
- `初始化配置`

对应后端模块：

- `metadata-config-module`
- `metadata-query-module`
- `addons/task-assistant`
- `addons/claude-worker-agent`

## 2. 功能范围

### 2.1 Git Provider 管理

- GitHub / GitLab / Gitee 配置
- Token 保存
- 编辑、删除、查询

### 2.2 LLM 模型管理

- 模型新增、编辑、删除
- 连通性测试
- 默认模型设置
- 模型重排
- 模型作用域限制
- Worker Backend 绑定

### 2.3 Agent 模型覆盖

- 为指定 Agent 绑定特定模型配置
- 删除覆盖关系

### 2.4 API 凭证管理

- 保存外部 API 凭证
- 选择认证方式
- 连接测试
- 分类管理

### 2.5 用户记忆管理

- 维护偏好、事实、备注等记忆
- 支持增删改查

### 2.6 Worker 治理

- 平台侧查看与管理 Claude Worker
- 健康检查
- 配置修改

### 2.7 任务助手

- 创建助手配置
- 绑定 Worker 与目录
- 选择模型
- 启停助手
- 触发测试通知

### 2.8 初始化向导

当前初始化向导是设置页的收敛版，分三步：

1. Git Provider
2. AI 模型
3. Worker

## 3. 设计特点

### 3.1 设置页是平台控制面，而不是偏好页

这里管理的多数内容会直接影响：

- 是否可执行任务
- 任务走哪个后端
- 用哪个模型和凭证
- 能否接入远程 Worker

### 3.2 配置体系已经形成分层

- Git 资源
- 模型资源
- 凭证资源
- 记忆资源
- Worker 资源
- Agent 级覆盖规则

### 3.3 初始化引导与长期治理共用同一套后端能力

这意味着：

- 初始化不是一次性脚本
- 后续治理和初始化复用同一配置模型

## 4. 典型使用场景

### 场景 1：平台首次可用

管理员先完成初始化向导，让系统具备最小运行条件。

### 场景 2：新增一套模型

管理员可以配置新的 LLM、测试连通性、指定默认或限制作用范围。

### 场景 3：为特定 Agent 固定模型

通过 Agent 模型覆盖，确保关键 Agent 使用特定模型而不受全局默认值影响。
