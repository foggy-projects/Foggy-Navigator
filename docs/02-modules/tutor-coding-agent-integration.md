# Tutor 与 Coding 能力联动

> 当前产品口径下，Tutor Agent 与编程执行能力之间的协作关系说明

## 1. 文档定位

本文档描述的是“引导型 Agent 如何把用户需求导向编程执行能力”，而不是某个单独 addon 的内部实现细节。

当前要点是：

- Tutor 负责识别意图、引导用户、路由到合适能力
- 真正的执行由 Worker / 任务分发 / addon 模块承担
- 这条链路当前不应再理解为“语义层 JavaScript 编辑流程”

## 2. 当前联动目标

当用户提出编程类、修改类、排查类需求时，系统需要完成三件事：

1. 判断这是不是普通对话，还是需要进入编程执行链路
2. 判断应该走哪种执行能力
3. 把用户带到正确的会话或任务上下文中

## 3. 当前参与角色

### 3.1 Tutor Agent

负责：

- 识别用户意图
- 提供引导式回答
- 在合适时发起委派或路由
- 帮用户衔接上下文

不负责：

- 直接管理远程目录
- 执行 Git/文件/终端操作
- 直接承担 Worker 执行过程控制

### 3.2 Session Module

负责：

- 承接会话入口
- 维护父子会话与路由跳转
- 统一任务分发
- A2A Agent 发现

### 3.3 编程执行能力

当前主要落在两类能力上：

- Claude Worker / Codex Worker 方向  
  适合远程目录、工作区、历史任务、文件/Git 上下文明确的执行。
- OpenHands 集成方向  
  适合作为独立集成模块存在，但不是当前主前端主入口。

## 4. 当前联动形态

### 4.1 会话内引导

典型路径：

```text
用户在会话中提问
  -> Tutor Agent 判断为编程类需求
  -> 给出引导或建议
  -> 路由到更合适的执行能力
  -> 用户进入目标会话或任务上下文
```

### 4.2 委派跳转

当前前端会话页已经支持：

- `DELEGATE`
- `RETURN`
- `NAVIGATE`

这意味着 Tutor 和其他 Agent 的联动，不是停留在文本建议，而是能落到真正的会话跳转。

### 4.3 Worker 工作台协作

当前主产品下，更常见的编程联动不是 Tutor 直接创建一个 OpenHands 会话，而是：

1. 用户在会话中被引导到编程执行方向
2. 再进入 Workers/目录上下文
3. 在目录中创建任务

## 5. 当前产品中的真实边界

### 5.1 当前已明确成立的部分

- Tutor 是统一会话入口的默认引导层
- 平台已支持委派和父子会话跳转
- 平台已支持通过统一任务分发进入执行链路
- Workers 工作台是当前主产品中的主要编程执行入口

### 5.2 当前不应再假设的部分

以下内容不能再当成默认事实：

- Tutor 直接主导完整 OpenHands 生命周期
- 所有编程任务都走 coding-agent
- 主产品的核心对象是 Git 凭证 + 语义层项目

这些更接近早期方案，而不是当前代码体现出来的主路径。

## 6. 推荐理解方式

当前更合理的理解是：

```text
Tutor
  -> 识别与引导
  -> 委派 / 路由
  -> 让用户进入正确执行上下文

执行上下文
  -> Session
  -> Worker
  -> Directory
  -> Task
```

也就是说，Tutor 更像“编程入口协调者”，而不是“编程执行引擎”。
