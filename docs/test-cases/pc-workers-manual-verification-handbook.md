# PC Workers 手工验证手册

## 1. 目标

这份手册用于你手动验证桌面端 `Workers` 页面本轮核心主链路，并重点覆盖最近暴露出的会话状态同步问题。

适用场景：

- 大改后快速手验
- 提交修复前的人工复现
- 提交修复后的回归确认

## 2. 测试前确认

先确认以下服务已启动：

| 项目 | 地址 | 期望 |
|------|------|------|
| 前端 | `http://localhost:5174` | 可打开登录页 |
| 后端 | `http://localhost:8112/actuator/health` | 返回 `UP` |
| Worker | `http://localhost:3031/health` | 返回正常 JSON |
| Proxy | `http://localhost:8082/health` | 返回 `healthy` |

登录账号：

- `root / root123`

建议使用一个已有目录，且该目录下已有少量历史会话，方便对比右侧列表刷新行为。

## 3. 最小执行顺序

按下面顺序执行，优先覆盖主链路：

1. 打开 `http://localhost:5174/#/login`
2. 登录系统
3. 进入 `Workers` 页面
4. 展开一个 `ONLINE` Worker
5. 选择一个可用目录
6. 确认目录头部展示正常
7. 新建一个简单会话
8. 观察右侧“历史会话”是否立即出现新卡片
9. 观察中间会话顶部状态与右侧卡片状态是否一致
10. 等待任务完成后再次确认状态是否一致
11. 继续该会话再发一轮，确认列表和状态仍同步

## 4. 详细验证项

### 4.1 页面进入与目录切换

检查项：

- `Workers` 页面可正常打开
- 左侧 Worker 树可展开
- 目录点击后，中间区域显示目录名、路径、Auth 标识

通过标准：

- 页面无明显白屏
- 目录切换后中间区域上下文正确

### 4.2 新建会话即时刷新

步骤：

1. 记住当前右侧历史会话列表最上方的第一条。
2. 输入一个简单 prompt，例如：`只回复 HAND_TEST_OK`
3. 点击发送/运行。

检查项：

- 中间会话区是否立即出现新 pane
- 右侧历史会话是否立即出现一个新卡片
- 新卡片是否位于预期位置，通常应靠前

通过标准：

- 不需要等任务完成，右侧就能看到新会话

失败判定：

- 必须等任务完成后，右侧才出现新卡片
- 必须手工点刷新后，右侧才出现新卡片

### 4.3 主会话与右侧卡片状态一致性

步骤：

1. 在新建任务后立刻观察中间会话区顶部动作条。
2. 同时观察右侧对应历史会话卡片。
3. 等待任务运行、完成，再各观察一次。

检查项：

- 处理中时，两边都体现为运行中/处理中
- 完成后，两边都切到可继续/待回复态
- 不应出现一边已完成，另一边仍显示处理中

通过标准：

- 主区域和右侧卡片在运行态、完成态都保持一致

失败判定：

- 主区域和右侧卡片状态文案或按钮含义不一致
- 右侧卡片明显滞后于中间主区域

### 4.4 继续会话回归

步骤：

1. 对刚完成的会话继续发送一轮。
2. 再次观察右侧历史会话状态变化。

检查项：

- 继续发送后，不会新增错误的重复卡片
- 同一会话卡片状态会重新切换到处理中
- 完成后状态再恢复

### 4.5 基础功能冒烟

可选补充确认：

- `同步会话`
- `浏览文件`
- `终端`
- `里程碑`
- 历史会话的 `继续`
- 历史会话的 `重连`

## 5. 本轮新增重点 BUG

本轮手验需要重点关注以下两个缺陷：

1. 新建会话后，右侧历史会话不自动刷新，要等任务完成才出现
2. 中间会话区域和右侧历史会话卡片状态不匹配

## 6. 建议记录方式

手测时建议每个异常都记录三类证据：

- 截图
- 出现时间
- 当前操作步骤

如果能补充浏览器控制台截图或 Network 中对应接口返回，会更利于后续定位。
