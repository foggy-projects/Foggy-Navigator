# Foggy Navigator 功能架构总说明

> 面向团队沟通、方案评审和新人理解的收官版说明。

## 1. 系统一句话定义

Foggy Navigator 当前是一套以 **多 Agent 协作、远程编程工作区、统一会话入口、任务治理与跨项目编排** 为核心的工作平台，而不是早期文档中描述的数据分析或语义层平台。

## 2. 系统解决什么问题

当前系统主要解决四类问题：

1. 让用户可以在统一入口中和不同 Agent 持续交互  
   包括 Tutor、Claude Worker、Codex Worker 以及通过 A2A 接入的其他执行能力。
2. 让编程执行不再脱离上下文  
   任务可以绑定到 Worker、目录、worktree、Git 状态和历史执行记录。
3. 让复杂工作流可以被治理  
   平台不是只有一次性问答，还可以查看任务、恢复任务、重连任务、跨项目分阶段推进任务。
4. 让平台具备可运营性  
   包括用户、凭证、模型、监控、通知、Open API 和 SDK。

## 3. 当前产品主轴

当前功能架构可以压缩成三条主轴：

### 3.1 会话主轴

会话是统一交互入口，负责承接用户提问、Agent 回复、委派跳转、分享与实时流式更新。

这条主轴回答的问题是：

- 用户和谁对话
- 对话如何持续
- 一个需求如何从当前 Agent 跳转到另一个执行上下文

### 3.2 Worker 主轴

Worker 是执行上下文中心，负责承接远程目录、文件、Git、终端和编程任务。

这条主轴回答的问题是：

- 任务在哪个环境里执行
- 任务绑定哪个目录、项目或分支
- 用户如何在执行现场查看和继续工作

### 3.3 任务主轴

任务是平台治理对象，负责把会话内触发的执行行为、Worker 中发起的目录任务，以及跨项目阶段任务统一起来。

这条主轴回答的问题是：

- 当前有哪些任务在运行
- 任务是否异常、中断、可恢复
- 复杂任务如何阶段化推进

## 4. 当前一级功能域

### 4.1 工作区与 Worker 中心

这是当前最重的业务操作台，聚焦远程 Worker、目录、文件、Git、终端和目录级任务执行。

### 4.2 会话协作中心

这是统一对话入口，聚焦消息流、Agent 路由、委派、父子会话和分享能力。

### 4.3 任务治理中心

这是平台治理入口，聚焦任务列表、状态、恢复、取消、重连和重同步。

### 4.4 跨项目编排

这是复杂任务管理入口，聚焦多阶段、多目录、多 Agent 的串行推进与 handoff。

### 4.5 平台设置与资源治理

这是平台运行配置入口，聚焦 Git、LLM、Worker、凭证、记忆、模型覆盖和任务助手配置。

### 4.6 用户与访问控制

这是管理员治理入口，聚焦登录、用户、角色和 API Key。

### 4.7 监控、通知与开放集成

这是平台可运营与可集成能力入口，聚焦监控事件、SSE、通知、Open API 和 SDK。

## 5. 当前核心角色分工

### 5.1 Tutor

Tutor 是默认引导层，负责识别需求、解释问题、做能力路由，并把用户导向正确的会话或执行上下文。

### 5.2 Worker 类执行能力

Claude Worker、Codex Worker 等能力负责在具体目录和执行环境里完成编程类任务，是当前主产品中的主要执行入口。

### 5.3 A2A 接入能力

A2A 负责把异构或外部 Agent 接入统一分发链路，是路由机制，不是单独的产品功能域。

### 5.4 编程执行入口

编程执行能力当前由 Claude、Codex、Gemini 等 Worker addon 承接，通过 Worker 目录、任务和 A2A 路由进入统一平台。

## 6. 当前边界判断

下面这些可以视为当前系统已经形成稳定口径的内容：

- 会话是统一交互入口
- Worker 是执行上下文中心
- 任务是平台治理对象
- 跨项目任务是复杂工作的阶段化编排方式
- 平台已经具备基础设置、监控、用户与开放集成能力

下面这些则不应再作为当前主线来描述：

- 数据分析平台
- 语义层管理平台
- 以 TM/QM 编辑为核心的产品定位
- 把整个系统直接等同为单一编程执行引擎的集成壳层

## 9. 结论

现在的 Foggy Navigator，本质上是一套围绕“会话入口 + Worker 执行现场 + 任务治理 + 跨项目编排”建立起来的多 Agent 工作平台。
