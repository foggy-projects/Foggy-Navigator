## 1. 业务背景和定位

### 1.1 项目定位

Foggy Navigator 是一个基于 AI 驱动的数据分析平台，核心价值在于：

- **自动化语义层生成**：通过 AI 自动分析数据库结构，生成语义层
- **智能引导**：通过"导师"Agent 引导管理员快速上手
- **灵活配置**：Agent 通过配置而非定制开发，支持灵活扩展
- **版本管理**：使用 Git 管理语义层文件，支持版本控制和回滚
- **沙盒机制**：提供安全的语义层测试和验证环境

### 1.2 核心价值

**对管理员**：
- 快速上手：导师 Agent 引导完成系统配置
- 自动化：AI 自动分析数据库，生成语义层
- 安全可控：沙盒环境 + Git 版本管理
- 智能提醒：定时监控数据库变化，主动提醒更新

**对用户**：
- 自然语言查询：基于语义层的 AI 数据分析
- 准确理解：语义层提供准确的业务含义
- 权限控制：基于语义层的精细化权限管理

## 2. 用户角色和职责

### 2.1 管理员（Admin）

**职责**：
- 配置数据源
- 管理语义层
- 配置权限
- 管理语义层版本
- 监控系统状态

**核心场景**：
1. 首次登录：导师 Agent 引导完成系统初始化
2. 添加数据源：配置数据库连接
3. 生成语义层：AI 自动分析数据库结构
4. 编辑语义层：手动调整和优化
5. 同步到正式环境：沙盒验证后发布
6. 配置权限：基于语义层配置数据访问权限

### 2.2 用户（User）

**职责**：
- 使用自然语言查询数据
- 查看数据分析结果
- 生成报表和可视化

**核心场景**：
1. 自然语言查询："上个月销售额是多少？"
2. 复杂查询："对比今年和去年同期的销售趋势"
3. 数据可视化：生成图表和报表
4. 导出结果：导出为 Excel、PDF 等格式

### 2.3 Agent（系统角色）

**导师 Agent（Tutor Agent）**
- 职责：引导管理员完成系统配置和任务
- 触发：管理员登录、系统配置检查
- 能力：检查配置、规划任务、推荐 Agent

**语义层生成 Agent（Semantic Layer Agent）**
- 职责：分析数据库结构，生成语义层
- 触发：管理员手动触发、定时任务触发
- 能力：读取 Schema、AI 分析、生成语义层

**数据分析 Agent（Data Analysis Agent）**
- 职责：理解用户查询，执行数据分析
- 触发：用户发起查询
- 能力：自然语言理解、SQL 生成、数据查询、结果可视化

**定时监控 Agent（Monitoring Agent）**
- 职责：监控数据库变化，提醒更新语义层
- 触发：定时任务
- 能力：Schema 对比、变化分析、生成更新建议

## 4. Agent 配置和管理

### 4.1 Agent 配置化设计

**核心原则**：Agent 通过配置文件定义，而非硬编码开发

**配置结构**：

```yaml
agent:
  id: "tutor-agent"
  name: "导师 Agent"
  description: "引导管理员完成系统配置和任务"
  type: "SYSTEM"
  
  capabilities:
    - name: "check_system_config"
      description: "检查系统配置"
      parameters:
        - name: "config_type"
          type: "string"
          required: true
          enum: ["datasource", "semantic", "permission"]
    
    - name: "plan_initialization"
      description: "规划初始化步骤"
      parameters:
        - name: "current_state"
          type: "object"
          required: true
    
    - name: "recommend_agent"
      description: "推荐合适的 Agent"
      parameters:
        - name: "task_type"
          type: "string"
          required: true
  
  triggers:
    - type: "user_login"
      enabled: true
      conditions:
        - key: "user_role"
          operator: "equals"
          value: "admin"
    
    - type: "system_check"
      enabled: true
      schedule: "0 */6 * * * *"  # 每6小时
  
  tools:
    - id: "check_datasource_config"
      type: "builtin"
      config:
        timeout: 30
    
    - id: "check_semantic_config"
      type: "builtin"
      config:
        timeout: 30
    
    - id: "query_database"
      type: "builtin"
      config:
        timeout: 60
  
  model:
    provider: "openai"
    model: "gpt-4"
    temperature: 0.7
    max_tokens: 2000
  
  memory:
    short_term:
      enabled: true
      max_messages: 50
    long_term:
      enabled: false
```

### 4.2 Agent 类型

**导师 Agent（Tutor Agent）**
- 类型：SYSTEM
- 触发：用户登录、系统检查
- 能力：配置检查、任务规划、Agent 推荐

**语义层生成 Agent（Semantic Layer Agent）**
- 类型：TASK
- 触发：手动触发、定时任务
- 能力：Schema 读取、AI 分析、语义层生成

**数据分析 Agent（Data Analysis Agent）**
- 类型：QUERY
- 触发：用户查询
- 能力：NLU、SQL 生成、数据查询、可视化

**监控 Agent（Monitoring Agent）**
- 类型：SCHEDULED
- 触发：定时任务
- 能力：Schema 对比、变化分析、通知

```yaml
semantic_layer:
  version: "1.0.0"
  created_at: "2024-01-01T00:00:00Z"
  created_by: "admin"
  
  entities:
    - name: "Customer"
      description: "客户信息"
      table: "customers"
      fields:
        - name: "id"
          type: "integer"
          description: "客户ID"
          primary_key: true
        
        - name: "name"
          type: "string"
          description: "客户姓名"
          searchable: true
        
        - name: "email"
          type: "string"
          description: "客户邮箱"
          searchable: true
        
        - name: "created_at"
          type: "datetime"
          description: "创建时间"
          format: "YYYY-MM-DD HH:mm:ss"
      
      relationships:
        - type: "one_to_many"
          target: "Order"
          foreign_key: "customer_id"
          description: "一个客户可以有多个订单"
    
    - name: "Order"
      description: "订单信息"
      table: "orders"
      fields:
        - name: "id"
          type: "integer"
          description: "订单ID"
          primary_key: true
        
        - name: "customer_id"
          type: "integer"
          description: "客户ID"
          foreign_key: "Customer.id"
        
        - name: "total_amount"
          type: "decimal"
          description: "订单总金额"
          format: "0.00"
        
        - name: "status"
          type: "string"
          description: "订单状态"
          enum: ["pending", "paid", "shipped", "cancelled"]
      
      relationships:
        - type: "many_to_one"
          target: "Customer"
          foreign_key: "customer_id"
          description: "一个订单属于一个客户"
  
  permissions:
    - role: "admin"
      entities:
        - name: "Customer"
          actions: ["read", "write", "delete"]
        - name: "Order"
          actions: ["read", "write", "delete"]
    
    - role: "user"
      entities:
        - name: "Customer"
          actions: ["read"]
        - name: "Order"
          actions: ["read"]
```

### 6.3 任务执行

**同步执行**：
- 顺序执行任务
- 等待前一个任务完成
- 失败时停止

**异步执行**：
- 并行执行独立任务
- 等待所有任务完成
- 失败时继续执行其他任务

**条件执行**：
- 根据前一个任务的结果决定是否执行
- 支持分支和循环

```
      └── 查看分析结果
```

**基于语义层的权限控制**：

```yaml
permissions:
  - role: "Data Analyst"
    entities:
      - name: "Customer"
        fields:
          - name: "id"
            actions: ["read"]
          - name: "name"
            actions: ["read"]
          - name: "email"
            actions: ["read"]
          - name: "phone"
            actions: []  # 无权限
      
      - name: "Order"
        actions: ["read"]
        conditions:
          - field: "status"
            operator: "equals"
            value: "paid"  # 只能查看已支付订单
```

- 用户引导和体验

- 定时监控和主动提醒

这个业务架构设计方案为后续的技术实现提供了清晰的指导。
